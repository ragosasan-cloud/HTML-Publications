Resumen del proyecto html-publisher

Documento de control interno · Desarrollos

Qué construimos

Una herramienta de línea de comandos en Python (publish.py) que automatiza la publicación de documentos HTML. Con un solo comando, hace dos cosas a la vez:

  1. Sube el archivo HTML a GitHub (al repo ragosasan-cloud/HTML-Publications), donde GitHub Pages lo sirve automáticamente como página web pública.
  2. Registra la publicación en una base de datos de Notion con el nombre del archivo, enlace en vivo, fecha, destinatario y carpeta.

El resultado: un enlace público tipo https://ragosasan-cloud.github.io/HTML-Publications/<carpeta>/<archivo>.html y un registro trazable en Notion.

Flujo de uso

  1. Ejecutas python3 publish.py.
  2. Te pregunta tres cosas: ruta al archivo HTML, carpeta (menú numerado de 1–5 o «Otro») y destinatario.
  3. Hace PUT al GitHub Contents API — si el archivo ya existe, lo actualiza usando el sha; si no, lo crea.
  4. Consulta Notion buscando una fila con el mismo nombre de archivo. Si existe, la actualiza (PATCH); si no, crea una nueva (POST).
  5. Imprime el enlace en vivo.

Configuración

  • .env guarda 4 secretos: GITHUB_TOKEN, GITHUB_REPO, NOTION_TOKEN, NOTION_DATABASE_ID.
  • Carpetas predefinidas: fiat-to-usdt-btc, monetizing-nano-copper, oro, teologia, venta-polvo-cobre.
  • Propiedades de la base Notion: Name (title), Live Link (url), Date (date), Recipient (rich_text), Folder (rich_text).

Bugs que arreglamos

  1. Sintaxis dict | None incompatible con Python 3.9 — añadimos from __future__ import annotations para que las anotaciones de unión funcionen en este Mac.
  2. Campo Folder vacío en Notion — la función build_properties no recibía ni enviaba folder. Lo añadimos como parámetro y como propiedad rich_text en el dict.
  3. Sin reintentos ante errores transitorios — un 502 Bad Gateway de Notion abortaba la publicación. Ahora hay reintentos con backoff exponencial vía urllib3.Retry para GET/PUT/PATCH, y una función publish_notion() que re-consulta Notion antes de cada intento para evitar filas duplicadas si un POST falló pero sí persistió.

Capacidades

  • Idempotente: ejecutar dos veces con el mismo nombre actualiza en vez de duplicar (tanto en GitHub como en Notion).
  • Crea o actualiza sin que el usuario tenga que decidir cuál es cuál.
  • Multi-carpeta: organiza publicaciones por tema (cinco categorías + cualquier nueva vía «Otro»).
  • Trazabilidad: cada publicación queda registrada en Notion con fecha y destinatario.
  • Auto-deploy: GitHub Pages publica el HTML automáticamente al hacer push, sin paso extra.
  • Compatible con Cowork Dispatch: como acepta stdin por tubería (printf '...' | python3 publish.py), Claude puede ejecutarlo en este Mac desde el iPhone.

Limitaciones

  • Interactivo por diseño: usa input(), no acepta argumentos de línea de comandos. Para integrar con webhooks, Shortcuts de iOS o cron, habría que añadir argparse (--file, --folder, --recipient).
  • Sin rollback de GitHub: si Notion sigue caído después de todos los reintentos, el archivo queda publicado en GitHub Pages aunque la fila de Notion no se cree. El script imprime un mensaje claro y sale con código 2; re-ejecutar es idéntico y registra la fila (idempotente).
  • Sin validación: no verifica que el archivo sea HTML válido, ni que la propiedad Name en Notion sea única.
  • Identificación por nombre de archivo: si publicas el mismo nombre en dos carpetas distintas, la fila de Notion se sobrescribirá con la última carpeta — no son únicas por (folder, filename).
  • GitHub Pages tiene latencia: el enlace puede dar 404 por ~30–60 segundos después de la primera publicación.
  • Sin soporte para subcarpetas anidadas dentro de cada categoría.
  • Sin borrado: no hay forma de despublicar un archivo desde el script (hay que borrarlo manualmente en GitHub y en Notion).
  • Token de GitHub con permisos amplios: el token tiene acceso de escritura al repo completo, no solo a una carpeta.
  • Python 3.9 en este Mac: funciona, pero futuras dependencias podrían requerir versiones más nuevas.

Próximas mejoras razonables

  • Añadir banderas de CLI para uso no interactivo (mejor para Dispatch, Shortcuts, automatización).
  • Hacer las filas de Notion únicas por (folder, filename) en lugar de solo filename.
  • Subcomando unpublish para borrar de GitHub + Notion en una sola operación.
  • Mensaje de error claro si GitHub Pages aún no está habilitado en el repo.
