Workflow de automatización — publish.py

Sube un HTML al repo ragosasan-cloud/HTML-Publications (GitHub Pages lo sirve) y registra la publicación en una base de datos de Notion. Idempotente: re-ejecutar con el mismo archivo actualiza en vez de duplicar. Con reintentos automáticos ante errores transitorios de Notion.

Detalle de cada nodo

01 · TRIGGER
Manual Trigger
El usuario invoca printf '...' | python3 publish.py. Tres líneas: filename, folder number (1–6), recipient.
02 · LOCAL FS
Read HTML File
Lee el archivo local con Path.read_bytes() y lo codifica en base64 para la API de GitHub.
03 · GITHUB API
GET sha (¿existe?)
Consulta /repos/{owner}/{repo}/contents/{folder}/{file}. Si responde 200, captura el sha.
04 · GITHUB API
PUT file
Sube el archivo. Si había sha, actualiza; si no, crea. GitHub Pages lo sirve en ~30–60 s.
05 · NOTION API
Query Database
POST /databases/{id}/query filtrando por Name = filename. Devuelve la fila si existe.
06 · BRANCH
IF row exists?
Si la query devuelve resultados → rama true (update). Si no → rama false (create).
07a · NOTION (TRUE)
PATCH row
PATCH /pages/{id} con las propiedades nuevas. Idempotente: re-publicar el mismo HTML no duplica filas.
07b · NOTION (FALSE)
POST row
POST /pages con Name, Folder, Recipient, Live Link, Date.
08 · OUTPUT
Live Link
Imprime https://ragosasan-cloud.github.io/HTML-Publications/<folder>/<file> en stdout.
Detalles del código real que el diagrama no muestra: reintentos con backoff exponencial (urllib3 Retry) en GET/PUT/PATCH ante 429/500/502/503/504; función publish_notion() que re-consulta antes de cada reintento para evitar filas duplicadas si un POST falló pero sí persistió; mensaje claro si GitHub publicó pero Notion sigue caído tras todos los intentos (re-ejecutar es idempotente).
